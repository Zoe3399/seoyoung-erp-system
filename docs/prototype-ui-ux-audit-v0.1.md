# 경남차유리 ERP 프로토타입 UI/UX 진단 v0.1

작성일: 2026-06-01  
대상: 현재 React 프로토타입, 요구사항 정의서 v0.2, Google Sheet 클라이언트 답변, 첨부자료 이미지, BOB0 실무 자료  
목적: 현재 프로토타입이 실제 업무에 최적화되지 않은 이유를 정리하고, 화면 재설계 방향을 잡기 위함

## 1. 총평

현재 프로토타입은 기능 항목은 많이 들어갔지만, 실제 업무자가 하루 업무를 처리하는 순서대로 설계되어 있지는 않다.

가장 큰 문제는 다음 세 가지다.

1. 화면이 업무 흐름보다 기능 묶음 중심으로 나뉘어 있다.
2. 입력 폼, KPI, 카드, 표가 같은 중요도로 섞여 있어 지금 무엇을 처리해야 하는지 잘 보이지 않는다.
3. 실무 핵심 데이터인 품목, 재고, 거래처, 단가, 결제, 문서 출력이 서로 연결된 운영 흐름으로 보이지 않는다.

즉, 지금 상태는 "ERP에 들어갈 수 있는 기능 데모"에 가깝고, "관리자가 실제로 매일 쓰는 업무 화면"으로는 다시 정리해야 한다.

## 2. 핵심 문제

### UX-001. 업무 흐름이 메뉴와 화면에 직접 드러나지 않는다

실제 업무 흐름은 다음에 가깝다.

견적 접수 -> 작업 예약 -> 작업 완료 -> 재고 차감 -> 청구/전표 생성 -> 입금/미수 확인

별도 판매 흐름은 다음과 같다.

상품 판매/매입 -> 재고 반영 -> 전표 생성 -> 입금/미수 확인

하지만 현재 화면은 홈, 매출, 판매관리, 견적, 작업, 일정, 단가표, 보증서, 청구, 장부, 재고, 고객, 거래처, 자료가 병렬로 흩어져 있다. 사용자는 "지금 이 건을 다음 단계로 넘기려면 어디로 가야 하는가"를 매번 판단해야 한다.

개선 방향:

| 현재 | 개선 |
| --- | --- |
| 기능별 메뉴 나열 | 업무 흐름별 메뉴 그룹 |
| 상단 메뉴와 사이드바 역할 혼재 | 상단은 핵심 업무 허브, 사이드바는 세부 메뉴 |
| 작업/정산/기준정보가 느슨하게 분리 | 작업 건 하나에서 견적, 작업, 재고, 보증서, 결제까지 연결 |

### UX-002. 화면마다 주인공이 불명확하다

현재 많은 화면에서 KPI 카드, 입력 폼, 목록, 보조 카드가 같은 레벨로 배치되어 있다. 운영 화면에서는 "오늘 처리해야 할 목록"이나 "예외 상태"가 주인공이어야 한다.

예시:

| 화면 | 현재 문제 | 개선 방향 |
| --- | --- | --- |
| 판매관리 | 등록/알림/목록/일괄등록이 한 화면에 모두 있어 시선이 분산됨 | 주문 처리 목록을 주인공으로 두고 등록/일괄등록은 보조 액션 |
| 단가표 | 출처별 통합표 느낌이 강함 | 유리/몰딩, 썬팅, 재단정보 탭 기반 기준정보 화면 |
| 보증서 | PDF 미리보기는 좋지만 전체 문서관리와 분리됨 | 문서 양식/도장/출력 이력을 통합 |
| 일정 | 일반 일정과 작업 일정의 목적이 섞일 수 있음 | 작업 캘린더와 결제 캘린더를 분리 |

### UX-003. 견적 화면이 실제 입력 화면의 밀도와 우선순위를 따라가지 못한다

클라이언트가 준 첨부자료의 견적 입력 화면은 아주 밀도 높은 운영형 폼이다. 빨간 박스로 표시된 항목들이 업무상 중요하다.

중요 항목:

- 지점/대리점
- 외주 여부
- 문의 경로
- 차량번호/소유자
- 차대번호
- 모델명
- 고객 연락처
- 수리구분
- 부품번호
- 옵션
- 썬팅
- 견적 내용
- 결제 정보
- 작업일/방문/출장/담당자

현재 프로토타입은 보기 좋은 모달/섹션 구조에 가깝지만, 실제 업무 화면처럼 빠르게 입력하고 누락을 확인하는 구조는 아직 부족하다.

개선 방향:

1. 견적 등록은 "간단 폼"보다 첨부자료 기반의 운영형 상세 폼으로 재구성한다.
2. 필수/선택을 시각적으로 나누되, 빈 값 임시저장은 허용한다.
3. 차량번호, 차종, 품번, 거래처는 한 화면에서 바로 검색/신규등록 가능하게 한다.
4. 견적에서 바로 작업등록으로 넘어가는 CTA를 명확히 둔다.

### UX-004. 작업 화면이 너무 많은 역할을 맡고 있다

현재 `WorkPage`는 코드 기준으로 매우 큰 화면이다. 작업 목록, 캘린더, 상세, 완료, 결제, 재고 차감, 일정 전환 등 많은 흐름이 한 컴포넌트에 몰려 있다.

문제:

- 작업자용과 관리자용 화면의 목적이 섞인다.
- 캘린더, 리스트, 상세, 등록의 우선순위가 명확하지 않다.
- 작업 완료 후 재고 차감, 청구/전표 생성, 보증서 발급이 하나의 후속 흐름으로 보이지 않는다.

개선 방향:

| 화면 | 목적 |
| --- | --- |
| 작업 캘린더 | 오늘/이번 주 작업 배정과 상태 확인 |
| 작업 리스트 | 검색, 필터, 일괄 상태 변경 |
| 작업 상세 | 견적, 부품, 재고, 결제, 보증서, 메모를 한 건 기준으로 확인 |
| 작업 완료 패널 | 부품 차감, 청구/전표 후보, 보증서 발급 여부 확인 |

### UX-005. 정산/결제 화면이 아직 업무 허브로 잡히지 않았다

대금결제 스케줄 참고 화면은 방향이 좋다. 이 화면처럼 결제는 별도 운영 대시보드가 필요하다.

필요한 구조:

- 기준월 이동
- 상태/거래처 필터
- 총 결제 예정액, 결제 완료액, 미결제/연체 KPI
- 결제 캘린더
- 결제 상세 리스트
- 인쇄/엑셀 다운로드
- 업무 바로가기
- 결제 업무 메모

현재 프로토타입에서는 정산 관련 기능이 장부, 카드매출, 대금결제, 매입, 미결로 흩어질 가능성이 있다. 정산 메뉴 안에서 "결제해야 할 것"과 "받아야 할 것"이 명확히 나뉘어야 한다.

### UX-006. 기준정보 화면이 실제 마스터 관리 형태로 정리되어야 한다

기준정보/단가관리 참고 화면이 훨씬 실무형에 가깝다.

필요한 기준정보 화면:

| 탭 | 목적 |
| --- | --- |
| 유리/몰딩 단가표 | 제조사, 차종, 구분, 옵션/사양, 부품가, 몰딩 정보 |
| 썬팅 필름 단가표 | 브랜드, 등급, 농도, 성능, 가격 |
| 차종별 재단 정보 | 제조사, 차종, 연식, 가로/세로, 비고 |
| 품목 마스터 | 5,030건 품목리스트 이관 |
| 재고 마스터 | 320건 재고리스트 이관 및 검증 |
| 거래처 마스터 | 164건 정비소/거래처 리스트 이관 |

현재 단가표 화면은 여러 출처를 한 표에 합친 형태라 빠른 프로토타입으로는 괜찮지만, 실무 관리 화면으로는 탭/필터/엑셀/페이지네이션 중심으로 다시 잡아야 한다.

### UX-007. 출력물/문서 기능은 별도 문서관리 개념이 필요하다

보증서만 독립적으로 구현하면 견적서, 거래명세서, 자동차 부품 대금 청구서와 흐름이 따로 놀 수 있다.

필요한 문서관리 개념:

- 문서 양식 마스터
- HTML 양식 등록
- 이미지 기준 양식 재현
- 필드 매핑
- 도장 마스터
- 도장 선택/위치 지정
- 문서별 기본 도장 위치
- PDF 미리보기/인쇄
- 발급/재발급 이력

특히 도장은 단순 이미지가 아니라 권한과 이력이 필요한 업무 데이터다.

### UX-008. 데이터 이관이 화면 경험으로 충분히 드러나지 않는다

Google Sheet에서 확인된 실제 데이터는 다음과 같다.

| 자료 | 건수 | 화면 영향 |
| --- | ---: | --- |
| 품목리스트 | 5,030건 | 품목 마스터, 단가, 작업 부품 검색 |
| 재고리스트 | 320건 | 재고 초기값, 입출고 이력, 부족 알림 |
| 거래처 리스트 | 164건 | 거래처 마스터, 견적/작업/정산 |
| 첨부자료 | 이미지 4개 | 견적 입력, 청구서, 견적서, 거래명세서 |

현재 프로토타입은 샘플 데이터 중심이라 실제 데이터 규모와 이관 검증 흐름이 잘 보이지 않는다.

개선 방향:

1. 자료 가져오기 화면을 "시트 목록 소개"가 아니라 "이관 상태 대시보드"로 바꾼다.
2. 품목/재고/거래처별로 총 건수, 정상, 오류, 중복, 미매칭을 보여준다.
3. 각 업무 화면에서 이관된 실제 데이터 일부를 검색 후보로 사용한다.

## 3. 정보구조 재설계안

기본 방향은 페이지를 과하게 나누는 것이 아니라, 실제 업무 한 건이 자연스럽게 다음 단계로 넘어가는 플로우를 만드는 것이다. 개별 데이터는 각 업무 화면의 목록, 검색, 상세 패널에서 접근하고, 기준정보는 별도 마스터 화면에서 관리한다.

### 3.1 플로우 중심 구조

| 플로우 | 흐름 | 화면 전환 방식 |
| --- | --- | --- |
| 작업 플로우 | 견적 접수 -> 작업 예약 -> 작업 완료 -> 재고 차감 -> 청구/전표 생성 -> 입금/미수 확인 | 한 건 상세 안에서 다음 단계 버튼으로 이동 |
| 판매 플로우 | 상품 판매/매입 -> 재고 반영 -> 전표 생성 -> 입금/미수 확인 | 판매 건 상세에서 출고/입고, 전표, 결제 상태를 이어서 처리 |
| 결제 플로우 | 매입/고정비 등록 -> 결제 예정 생성 -> 결제 완료/연체 확인 -> 장부 반영 | 대금결제 운영 화면에서 월별로 처리 |
| 문서 플로우 | 견적/작업/판매 데이터 선택 -> 문서 생성 -> 도장 선택 -> PDF/인쇄 -> 발급 이력 | 각 업무 상세에서 문서 생성, 문서관리에서 이력 조회 |

### 3.2 데이터 목록 구조

각 데이터는 독립 페이지로 흩어놓기보다, 업무 흐름 안에서 목록과 상세로 확인할 수 있어야 한다.

| 데이터 | 목록이 필요한 위치 | 상세에서 해야 할 일 |
| --- | --- | --- |
| 견적 | 견적 접수 화면 | 작업 등록, 견적서 출력, 고객/차량 확인 |
| 작업 | 작업 캘린더/작업 리스트 | 완료 처리, 재고 차감, 보증서/청구/결제 연결 |
| 판매 | 판매관리 화면 | 출고/입고, 재고 차감, 거래명세서, 결제 확인 |
| 대금결제 | 대금결제 화면 | 결제 완료, 연체 확인, 매입 상세 이동 |
| 품목 | 품목/재고 화면, 견적/작업 품목 검색 | 단가, 재고, 사용 이력 확인 |
| 거래처 | 견적/작업/정산 검색, 거래처 목록 | 미결, 결제, 작업 이력 확인 |
| 문서 | 업무 상세, 문서관리 화면 | PDF 보기, 도장 위치, 재발급 |

추천 메뉴 구조:

| 1차 메뉴 | 2차 메뉴 | 목적 |
| --- | --- | --- |
| 홈 | 오늘 할 일 | 작업/견적/정산 예외 요약 |
| 견적/작업 | 견적 접수 | 첨부자료 기반 운영형 견적 입력 |
| 견적/작업 | 작업 캘린더 | 작업 배정과 상태 관리 |
| 견적/작업 | 작업 리스트 | 검색, 필터, 일괄 처리 |
| 판매/재고 | 판매관리 | 상품 판매, 매입입고, 출고, 결제 |
| 판매/재고 | 품목 마스터 | 품목리스트 5,030건 관리 |
| 판매/재고 | 재고관리 | 입고, 출고, 조정, 부족 알림 |
| 정산 | 대금결제 | 월별 결제 스케줄 |
| 정산 | 카드매출 | 카드 입금/수수료 |
| 정산 | 매입/미결 | 매입, 미수, 미지급 |
| 보험/청구 | 청구 계산 | 보험사별 청구 산식 |
| 기준정보 | 단가관리 | 유리/몰딩, 썬팅, 재단 정보 |
| 기준정보 | 거래처 | 정비소/거래처 마스터 |
| 문서 | 출력물 | 견적서, 거래명세서, 청구서, 보증서 |
| 문서 | 도장/양식 | 도장, 템플릿, 기본 위치 |
| 시스템 | 자료 이관 | 시트 가져오기/검증 |
| 시스템 | 권한/로그 | 사용자, 보안, 변경 이력 |

## 4. 화면 패턴 재정의

운영 화면은 다음 패턴을 기본으로 둔다.

1. 상단: 현재 화면 목적 + 핵심 액션 1개
2. 요약: 반드시 필요한 KPI 3개 이하 또는 예외 요약
3. 중심: 리스트/캘린더/작업 보드
4. 우측 또는 하단: 선택 건 상세/후속 액션
5. 등록: 드로어 또는 별도 상세 화면
6. 출력/엑셀: 상단 보조 액션

피해야 할 패턴:

- 입력 폼이 목록보다 먼저 나오는 화면
- KPI 카드가 너무 많아지는 화면
- 설명성 카드가 실제 업무 데이터를 밀어내는 화면
- 한 화면에 등록, 목록, 도움말, 흐름 설명, 통계가 모두 같은 비중으로 있는 구조
- 탭과 필터가 모바일/중간 해상도에서 세로로 무너지는 구조

## 5. 재설계 우선순위

### P0. 먼저 다시 잡아야 할 화면

## 6. 결론

현재 프로토타입은 계속 기능을 덧붙이기보다, 핵심 업무 화면 5개를 먼저 다시 설계하는 것이 맞다.

추천 작업 순서:

1. 메뉴/정보구조를 업무 흐름 기준으로 재정리한다.
2. 견적 접수 화면을 첨부자료 기준으로 재설계한다.
3. 작업 상세를 중심으로 견적, 재고, 보증서, 결제 후속 흐름을 연결한다.
4. 판매관리와 대금결제를 각각 운영 보드 형태로 재구성한다.
5. 기준정보/단가관리 화면을 탭, 필터, 엑셀, 페이지네이션 중심으로 재작성한다.

한 줄로 정리하면, 지금은 "예쁘게 기능이 많은 ERP"보다 "실무자가 다음 처리 건을 놓치지 않는 ERP"로 방향을 바꿔야 한다.

## 7. 플로우 중심 화면 설계 원칙

페이지를 많이 나누는 대신, 사용자가 한 업무 건을 따라가며 자연스럽게 다음 단계로 넘어갈 수 있어야 한다.

원칙:

1. 견적, 작업, 판매, 결제는 각각 독립 메뉴이면서도 상세 화면에서는 다음 단계로 바로 전환된다.
2. 목록 화면은 데이터를 찾는 곳이고, 상세 화면은 업무를 처리하는 곳이다.
3. 기준정보는 평소에는 검색/추천으로만 노출하고, 필요할 때만 마스터 화면에서 관리한다.
4. 문서 출력, 도장, 결제, 재고 차감은 별도 페이지로 사용자를 보내기보다 현재 업무 상세 안에서 시작한다.
5. 다만 전체 데이터 점검이 필요한 경우를 위해 각 데이터별 목록은 유지한다.
6. 입력 폼은 기본 목록 화면에 상시 노출하지 않고, 명확한 버튼 액션으로 모달 또는 드로어에서 처리한다.

예시:

| 현재 방식 | 개선 방식 |
| --- | --- |
| 견적 저장 후 사용자가 작업 메뉴로 이동 | 견적 상세에서 `작업 예약` 버튼 클릭 |
| 작업 완료 후 재고 메뉴로 이동 | 작업 완료 패널에서 사용 부품 확인 후 재고 자동 차감 |
| 보증서 메뉴에 따로 들어가 발급 | 작업 상세에서 `보증서 발급` 클릭, 보증서 목록에서는 이력 조회 |
| 판매 후 장부 메뉴로 이동 | 판매 상세에서 `전표 생성`과 `입금 확인` 처리 |
| 대금결제와 매입이 분리되어 왕복 | 대금결제 상세에서 관련 매입 내역을 같이 표시 |

## 8. 입력 UI 원칙

기본 목록 화면은 보기, 검색, 필터, 상태 확인, 다음 처리 액션에 집중한다. 신규 등록이나 상세 입력은 버튼을 눌렀을 때 모달 또는 드로어로 열리게 한다.

기본 원칙:

| 상황 | 권장 UI | 예시 |
| --- | --- | --- |
| 짧은 신규 등록 | 모달 | 결제 등록, 판매 등록, 단가 등록 |
| 긴 신규 등록 | 큰 모달 또는 우측 드로어 | 견적 등록, 작업 예약 |
| 상세 수정 | 우측 드로어 | 견적 상세 수정, 작업 상세 수정, 거래처 상세 |
| 확인/완료 처리 | 확인 모달 | 작업 완료, 재고 차감, 결제 완료 |
| 출력 미리보기 | 큰 모달 | 견적서, 거래명세서, 청구서, 보증서 PDF |
| 도장 위치 조정 | 미리보기 모달 | 도장 선택, 위치 이동, 크기 조절 |

화면별 적용 예시:

| 화면 | 기본 화면 | 버튼 액션 |
| --- | --- | --- |
| 견적 목록 | 견적 검색, 상태 필터, 다음 액션 | `+ 견적 등록` -> 견적 등록 모달 |
| 견적 상세 | 견적 정보, 품목, 금액, 상태 | `작업 예약` -> 작업 예약 모달 |
| 작업 상세 | 작업 일정, 부품, 결제, 보증서 상태 | `작업 완료` -> 완료 처리 모달 |
| 판매관리 | 판매/매입 목록, 출고/입고 상태 | `+ 판매 등록` -> 판매 등록 모달 |
| 대금결제 | 월별 결제 목록, 캘린더, KPI | `+ 결제 등록` -> 결제 등록 모달 |
| 단가관리 | 단가 목록, 탭, 필터 | `+ 신규 등록` -> 단가 등록 모달 |
| 문서관리 | 문서 목록, 발급 이력 | `도장 선택` -> 도장 위치 조정 모달 |

피해야 할 방식:

- 목록 화면 상단에 큰 입력 폼을 항상 노출
- 등록 폼 때문에 목록과 상태가 아래로 밀리는 구조
- 한 화면에서 등록, 목록, 도움말, 통계를 모두 같은 비중으로 노출
- 저장과 다음 단계 이동 버튼이 화면 곳곳에 흩어지는 구조
