# 배포 및 운영 비용 전략

## 목적

본 문서는 서영 ERP 시스템의 MVP 개발 과정에서 배포 방식과 운영 비용을 함께 검토한 의사결정 기록이다.

단순히 기능 구현에 집중하는 것이 아니라, 실제 소규모 사업장에서 지속적으로 사용할 수 있는 수준의 운영 비용, 유지보수 난이도, 확장 가능성을 함께 고려했다. 특히 초기 예산이 크지 않은 상황을 전제로 월 운영비를 10만원 미만으로 제한하고, 서비스가 성장할 때 단계적으로 확장할 수 있는 구조를 목표로 했다.

## 프로젝트 전제

현재 시스템은 다음과 같은 구조를 가진다.

```text
apps/web
  React + TypeScript + Vite 기반 프론트엔드

apps/api
  NestJS + TypeScript 기반 백엔드 API

packages/database
  PostgreSQL + Prisma 기반 데이터베이스 계층
```

ERP 특성상 고객, 차량, 견적, 작업, 결제, 재고, 문서 이력 등 업무 데이터가 누적된다. 따라서 단순 정적 웹사이트 배포보다 서버와 데이터베이스의 안정성이 중요하다.

## 비용 제약 조건

초기 운영 단계에서는 다음 조건을 기준으로 배포 방식을 검토했다.

| 항목 | 기준 |
|---|---|
| 월 운영비 | 10만원 미만 |
| 초기 목표 | MVP 운영 및 실제 업무 검증 |
| 사용자 규모 | 소규모 내부 사용자 중심 |
| 서버 구성 | 프론트엔드, API, DB 운영 가능 |
| 운영 우선순위 | 비용 예측 가능성, 백업 가능성, 확장 여지 |

## 검토한 배포 방식

### 1. 관리형 서비스 조합

```text
Vercel 또는 Cloudflare Pages
  + Supabase 또는 Neon
  + Render, Railway, Fly.io
```

장점은 배포가 쉽고 초기 설정이 빠르다는 점이다. GitHub에 push하면 자동 배포가 가능하고, 프론트엔드와 DB를 빠르게 연결할 수 있다.

다만 ERP처럼 API 서버와 DB가 함께 필요한 시스템에서는 서비스가 여러 개로 나뉘면서 비용 구조가 복잡해질 수 있다. 무료 플랜으로 시작할 수는 있지만, 프로젝트가 늘어나거나 사용량이 증가하면 각 서비스의 과금 조건을 별도로 관리해야 한다.

### 2. 단일 VPS 기반 배포

```text
Ubuntu VPS 1대
  + Docker Compose
  + Nginx 또는 Caddy
  + PostgreSQL
  + NestJS API
  + React 정적 파일
```

단일 VPS 방식은 서버 관리 부담이 일부 있지만, 월 비용을 예측하기 쉽고 프론트엔드, API, DB를 한 서버에서 운영할 수 있다. MVP 단계의 ERP처럼 트래픽이 크지 않고 내부 사용자가 중심인 시스템에는 비용 대비 효율이 높다.

## 권장 초기 구성

본 프로젝트의 초기 운영 구성은 다음과 같이 설계한다.

```text
사용자
  -> 도메인
  -> Nginx 또는 Caddy
  -> React 정적 파일
  -> NestJS API
  -> PostgreSQL
```

예상 비용은 다음과 같다.

| 항목 | 예상 월 비용 |
|---|---:|
| VPS 2GB RAM급 | 약 7천원 ~ 2만원 |
| 도메인 | 월 환산 약 1천원 ~ 2천원 |
| 백업 스토리지 | 약 0원 ~ 1만원 |
| 모니터링/기타 | 약 0원 ~ 5천원 |
| 합계 | 약 1만원 ~ 3만 7천원 |

초기에는 2GB RAM급 VPS로 시작하고, 사용자 증가나 데이터 처리량 증가가 확인되면 4GB 이상으로 확장한다. 이 방식은 월 10만원 미만의 예산 안에서 충분히 운영 가능하며, 실제 사용량이 검증되기 전 과도한 인프라 비용을 피할 수 있다.

## PM 관점의 설계 반영

이번 배포 전략은 단순한 개발 편의가 아니라 제품 운영 관점의 제약을 반영한 결정이다.

- 초기 고객 검증 단계에서는 고정비를 낮춰 제품 지속 가능성을 확보한다.
- ERP 데이터는 장기 보관이 중요하므로 DB 백업 정책을 초기부터 고려한다.
- API와 DB를 분리 가능한 구조로 유지해 향후 관리형 DB 또는 별도 API 서버로 이전할 수 있게 한다.
- 사용자가 적은 단계에서는 단일 서버 구성을 사용해 비용과 운영 복잡도를 낮춘다.
- 서비스 확장 시에는 프론트엔드, API, DB를 단계적으로 분리할 수 있도록 모노레포 구조를 유지한다.

## 단계별 확장 계획

### 1단계: MVP 검증

```text
VPS 1대
  - React 정적 파일
  - NestJS API
  - PostgreSQL
  - 자동 HTTPS
  - 정기 DB 백업
```

목표는 실제 업무 흐름 검증이다. 견적, 작업, 재고, 전표, 결제 상태 관리가 실제 현장에서 문제없이 돌아가는지 확인한다.

### 2단계: 운영 안정화

```text
VPS 유지
  - 백업 자동화
  - 로그 관리
  - 장애 알림
  - 관리자 계정 및 권한 정리
```

업무 데이터가 쌓이기 시작하면 백업과 권한 관리가 핵심이 된다. 이 단계에서는 새로운 기능보다 데이터 손실 방지와 운영 안정성을 우선한다.

### 3단계: 서비스 확장

```text
Frontend: Cloudflare Pages 또는 Vercel
API: VPS, Render, Railway 중 선택
DB: PostgreSQL 관리형 서비스 또는 별도 DB 서버
Storage: 문서/PDF 파일 저장소 분리
```

트래픽, 파일 저장량, 외부 연동이 증가하면 프론트엔드, API, DB를 분리한다. 이 시점에는 비용보다 장애 격리, 백업, 보안, 배포 자동화가 더 중요해진다.

## 운영 리스크와 대응

| 리스크 | 대응 방향 |
|---|---|
| VPS 장애 | 주기적 DB 백업, 서버 재구성 문서화 |
| DB 데이터 손실 | 일 단위 자동 백업, 백업 복구 테스트 |
| 서버 보안 | SSH 키 인증, 방화벽, HTTPS 적용 |
| 비용 증가 | 월 비용 모니터링, 리소스 증설 기준 수립 |
| 서비스 확장 한계 | API/DB 분리 가능한 구조 유지 |

## 의사결정 요약

초기 ERP 시스템은 많은 사용자를 대상으로 한 대규모 서비스가 아니라, 실제 업무 프로세스를 안정적으로 디지털화하는 것이 목적이다. 따라서 초기에는 관리형 서비스를 여러 개 조합하기보다 단일 VPS 기반으로 비용을 낮추고, 필요한 시점에만 인프라를 분리하는 전략이 적합하다.

이 결정은 월 10만원 미만의 비용 제약을 만족하면서도, MVP 검증, 운영 안정화, 향후 확장까지 고려한 현실적인 배포 설계이다.
