# 저렴한 배포 방향

## 현재 프로토타입 단계

DB/API 없이 화면만 보여주는 단계라면 정적 배포가 가장 싸고 관리가 쉽습니다.

추천:

- 1순위: Cloudflare Pages
- 2순위: Vercel

배포 설정:

- Root directory: 저장소 루트
- Build command: `npm run build`
- Output directory: `apps/web/dist`

이 방식은 서버를 직접 관리하지 않아도 되고, GitHub에 푸시하면 자동으로 새 화면을 배포할 수 있습니다.

## MVP 내부 사용 단계

한 가게에서 실제로 쓰는 ERP라면 처음부터 서버를 크게 잡을 필요는 없습니다.

추천 구성:

- Frontend: Cloudflare Pages
- DB/Auth/File: Supabase
- API: 처음에는 생략하거나, 필요할 때만 작은 서버로 분리

이 구성이 좋은 이유:

- 월 고정비를 낮게 시작할 수 있음
- 로그인, DB, 파일 저장을 빠르게 붙일 수 있음
- 나중에 API 서버를 붙여도 구조 변경 부담이 작음

주의:

- 실제 업무 데이터가 들어가면 무료 DB만 믿고 운영하기보다는 백업/중단 정책을 확인해야 합니다.
- 세금계산서, 고객 연락처, 차량 정보, 보험 정보가 들어가면 접근 권한과 이력 관리가 필요합니다.

## API 서버가 필요한 단계

보험 청구 자동화, 파일 처리, 외부 자료 수집, 문자/카카오 알림, 회계 연동이 들어가면 API 서버가 필요해집니다.

저렴한 후보:

- Railway: 작은 API 서버를 빠르게 올리기 좋음
- Render: 테스트/프리뷰에는 무료도 가능하지만 운영용은 유료 권장
- Fly.io: 사용량 기반이라 작게 시작 가능하지만 비용 관리 확인 필요
- 국내 VPS: 트래픽이 국내 중심이고 서버 관리가 가능하면 장기적으로 예측 가능

## CI/CD는 언제 할까

지금처럼 프론트엔드 프로토타입만 있는 단계에서는 복잡한 CI/CD가 꼭 필요하지 않습니다.

우선은 아래 정도면 충분합니다.

- 로컬: `npm run check` 또는 Docker의 `docker compose up`
- 공유/배포: Cloudflare Pages에서 GitHub push 시 자동 빌드

백엔드와 DB가 붙으면 그때 CI/CD를 더 엄격하게 잡습니다.

- API 타입 체크와 테스트
- DB migration 검증
- 운영 환경변수 검증
- 스테이징 배포 후 운영 배포
- 배포 실패 시 롤백

즉, 지금은 화면 검증 속도가 더 중요하고, 백엔드가 생기는 시점부터 배포 자동화를 정교하게 만드는 게 좋습니다.

## 추천 결론

지금은 아래 순서가 가장 현실적입니다.

1. `npm run dev`로 로컬 프로토타입 검증
2. Cloudflare Pages에 화면만 배포해서 클라이언트 피드백
3. 확정된 화면 기준으로 Supabase DB 설계
4. 실제 운영 전에 백업, 권한, 첨부파일 저장 정책 확정
5. 자동화가 많아지는 시점에 API 서버 추가
