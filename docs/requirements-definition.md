# 경남차유리 ERP 요구사항 정의서

문서 버전: v0.1 초안  
작성일: 2026-05-28  
대상 프로젝트: 경남차유리 업무관리 ERP  
작성 목적: 초기 요구사항, 기존 자료 분석, 프로토타입 설계, 클라이언트 피드백을 통합해 MVP 개발 범위와 우선순위를 확정하기 위한 기준 문서

## 1. 문서 개요

### 1.1 배경

경남차유리는 현재 견적, 작업내용, 작업 스케줄을 구글시트에 수기로 입력하고 관리하고 있다. 이 방식은 초기에는 빠르게 시작할 수 있지만, 업무량이 늘어날수록 다음 문제가 발생한다.

- 견적과 작업 일정이 분리되어 있어 같은 정보를 반복 입력한다.
- 작업 일정, 입금 확인, 보험 청구, 재고 확인 같은 후속 업무가 한 화면에서 이어지지 않는다.
- 차량, 고객, 거래처, 부품, 보험사 정보를 다시 찾는 데 시간이 걸린다.
- 카드매출, 대금결제, 보증서, 장부 정산처럼 별도 시트나 수기 계산에 의존하는 영역이 남아 있다.
- 작업이 끝난 뒤 사진, 사용 부품, 청구, 입금, 미수금까지의 연결이 약하다.

따라서 본 시스템은 단순한 일정표가 아니라, 한 매장에서 매일 반복되는 견적 접수, 작업 예약, 작업 완료, 재고 차감, 청구, 입금, 정산 업무를 하나의 흐름으로 처리하는 실무형 ERP를 목표로 한다.

### 1.2 시스템 목표

한 문장으로 정리하면 다음과 같다.

> 구글시트 없이 견적 접수부터 작업 일정, 작업 완료, 재고 차감, 청구/전표 생성, 입금/미수 확인까지 기본 사무업무가 한 시스템 안에서 이어지게 만든다.

핵심 목표는 다음과 같다.

- 내부 직원이 빠르게 로그인하고 오늘 처리할 업무를 확인한다.
- 견적을 짧은 필수 정보만으로 빠르게 저장하고, 나중에 상세 정보를 보완할 수 있다.
- 견적이 확정되면 작업으로 바로 전환하고 캘린더에 자동 표시한다.
- 작업 없이 바로 등록되는 작업과 일반 할 일도 같은 일정 화면에서 관리한다.
- 작업 완료 시 사진, 사용 부품, 재고 차감, 청구/전표 생성 흐름을 이어준다.
- 보험 청구, 고객 결제, 카드매출, 입금, 미수금을 분리해서 추적한다.
- 고객, 차량, 거래처, 보험사, 정비소, 부품 정보를 검색형 선택 방식으로 빠르게 찾는다.
- 향후 품목/재고, 전표, 보증서, 문서, 엑셀 업로드, 알림까지 확장 가능한 구조를 둔다.

### 1.3 문서 작성 기준

본 문서는 다음 자료를 종합했다.

- 최초 요구사항 정의서: 서영 ERP 시스템 요구사항, 견적/작업 스케줄 관리 중심
- 기존 참고 시스템 분석: 글로벌 ERP, ECOUNT 참고, 단 직접 모방하지 않고 업무 필드와 구조만 참고
- 프로젝트 작업 로그: 브랜드 변경, 로그인 편의 기능, 작업 캘린더, 상세 패널, 공통 UI 패턴 등 현재까지 작업 내용
- MVP 및 MVP V2 설계 문서: 단일 매장 기준 ERP 범위, 데이터 모델, 화면 우선순위
- 사용자 Flow 문서: 로그인, 홈, 견적, 작업 전환, 캘린더, 일반 할 일, 작업 상세 흐름
- 프로토타입 체크리스트: 견적/작업/청구/재고/자료/설정 우선순위
- 2026-05-28 Notion 작업 피드백: 사이드바, 대금결제리스트, 보증서, 카드매출 화면 신규/수정 요청

민감 정보 처리 원칙: 원본 자료에 포함된 외부 시스템 계정, 비밀번호, 개인 연락처, 주소 등 민감 정보는 본 문서에 재기재하지 않는다. 개발과 테스트에는 별도 샘플 데이터 또는 마스킹 데이터를 사용한다.

## 2. 프로젝트 범위

### 2.1 MVP 포함 범위

MVP는 단일 매장 내부 사용을 기준으로 한다. 여러 지점, 여러 법인, 외부 고객 포털은 기본 범위에서 제외한다.

- 로그인 기본 구조와 ID 저장, 자동로그인 편의 기능
- 오늘 업무 중심 대시보드
- 견적 리스트, 견적 등록, 견적 상세, 견적 상태 관리
- 견적에서 작업으로 바로 등록하는 흐름
- 작업 직접 등록, 작업 리스트, 작업 캘린더, 작업 상세
- 작업 외 일반 할 일 등록과 완료 처리
- 작업 완료 시 사진, 사용 부품, 재고 차감, 청구/전표 연결
- 고객, 차량, 거래처, 정비소, 보험사, 보험 담당자 관리
- 부품 마스터, 실재고, 입출고 이력
- 청구 리스트, 다이렉트 청구, 보험 입금 확인
- 전표, 결제, 미수금 관리
- 카드매출 정산 화면
- 대금결제리스트와 반복 지급 스케줄 기초 구조
- 보증서 목록, 등록, 발행, 문서 연결 구조
- 자료 첨부와 업무별 파일 연결
- 엑셀 업로드/다운로드 기초 구조
- 주요 변경 이력과 감사 로그

### 2.2 MVP 제외 범위

다음은 고도화 또는 추후 협의 범위로 둔다.

- 다중 회사가 같은 시스템을 독립적으로 쓰는 멀티테넌트 구조
- 여러 지점의 권한, 정산, 재고를 완전히 분리하는 조직 관리
- 모바일 기사 전용 앱
- 보험사 API 직접 연동
- 회계 프로그램 자동 연동
- 전자세금계산서 자동 발행
- 바코드/QR 기반 입출고
- 카카오 알림톡 실제 발송 API 연동
- 범용 문서함, 사용자 지정 폴더, 외부 자료 자동 수집함
- OCR 기반 세금계산서/청구서 자동 인식

### 2.3 시스템 형태

- 1차 목표 형태: 웹 시스템
- 실행 환경: 크롬 또는 엣지 브라우저
- 사용자: 경남차유리 내부 직원
- 예상 동시 접속자: 약 5명
- 현재 개발 방식: React + TypeScript + Vite 기반 프론트엔드 프로토타입
- 향후 개발 방식: NestJS API, PostgreSQL, Prisma 기반 백엔드/DB 연결

## 3. 사용자와 권한

### 3.1 사용자 유형

| 사용자 유형 | 설명 | MVP 접근 범위 |
|---|---|---|
| 관리자 | 매출, 청구, 재고, 장부, 설정을 관리하는 사용자 | 전체 업무 메뉴 조회/수정 |
| 직원 | 견적, 작업, 고객/차량, 사진 첨부를 주로 처리하는 사용자 | 견적/작업 중심 조회/수정 |
| 알바/단기 사용자 | 특정 작업 확인과 사진 첨부 위주 사용자 | 제한된 작업만 조회/수정 |
| 조회 전용 | 업무 현황 확인만 하는 사용자 | 조회만 가능 |

### 3.2 권한 원칙

- 일반 사용자의 회원가입은 제공하지 않는다.
- 계정 생성과 권한 변경은 최고 관리자 기능으로 고도화 단계에서 제공한다.
- MVP에서는 로그인 UX와 role 기반 접근 구조만 먼저 둔다.
- 프론트 메뉴 숨김만으로 권한을 처리하지 않고, API에서도 권한을 검사한다.
- 퇴사자, 단기 사용자 계정은 삭제보다 비활성화 상태로 관리한다.
- 민감 정보는 역할에 따라 마스킹한다.

## 4. 핵심 업무 흐름

### 4.1 전체 업무 흐름

1. 문의 접수 또는 견적 등록
2. 고객, 차량, 거래처, 작업내용을 가능한 범위에서 입력
3. 견적 상태를 대기, 보류, 확정, 취소로 관리
4. 확정된 견적은 작업으로 바로 전환
5. 작업일, 시간, 방문/출장, 담당자를 지정
6. 작업 캘린더와 오늘 할 일에 자동 표시
7. 작업 상세에서 사진, 사용 부품, 메모를 확인
8. 작업 완료 시 사용 부품을 확정하고 재고 출고 이력 생성
9. 보험/정비소 청구 또는 고객 결제 전표 생성
10. 입금, 카드매출, 미수금, 대금결제 현황 관리
11. 필요한 경우 보증서, 견적서, 청구서, 자료 첨부를 연결

### 4.2 견적에서 작업 전환 흐름

- 견적 상세에서 `작업으로 등록` 버튼을 제공한다.
- 작업 전환 시 견적의 고객명, 연락처, 차량정보, 작업내용, 금액, 부품, 보험 관련 메모를 자동으로 가져온다.
- 작업 전환 화면에서는 작업일, 작업시간, 방문/출장, 담당자 확인을 먼저 보여준다.
- 작업 저장 후 캘린더와 작업 리스트에 자동 표시한다.
- 사용자는 견적이 작업으로 넘어갔다는 상태를 바로 확인할 수 있어야 한다.

### 4.3 작업 완료 흐름

- 작업자는 작업 상세에서 사진과 사용 부품을 확인한다.
- 작업 완료 버튼을 누르면 사용 부품 확인, 재고 차감, 청구/전표 생성 후보가 표시된다.
- 재고 차감과 청구 확정은 자동 실행보다 사용자 확인 후 반영한다.
- 완료 후에는 작업 상태, 재고 이력, 청구/전표 연결 상태가 함께 갱신되어야 한다.

### 4.4 정산 흐름

- 보험 청구금액, 보험 입금액, 고객 결제금액은 분리해서 본다.
- 카드 결제는 카드매출 화면에서 입금일자, 입금금액, 수수료율을 관리한다.
- 반복 지급 또는 수기 결제 일정은 대금결제리스트에서 기준월/거래처별로 관리한다.
- 입금이 여러 번 나뉘어 들어올 수 있으므로 Payment와 PaymentAllocation 구조를 고려한다.
- 미수금은 청구합계 또는 매출합계에서 입금합계를 뺀 값으로 계산한다.

## 5. 정보 구조와 메뉴

### 5.1 기본 정보 구조

1차 메뉴는 현장 사용자가 이해하기 쉬운 업무 단어를 사용한다.

| 1차 메뉴 | 2차 메뉴 예시 | 비고 |
|---|---|---|
| 홈 | 오늘 업무, 업무 바로가기 | 대시보드 |
| 작업 | 견적, 작업, 일정, 청구, 보증서 | 견적-작업-청구 중심 |
| 판매/구매 | 판매관리, 구매관리, 대금결제리스트, 재고/입고 | 상품/구매 흐름 |
| 정산 | 매출, 장부, 카드매출, 미수금 | 금액 확인과 정산 |
| 기준정보 | 고객/차량, 거래처, 정비소, 보험사/담당자, 차종 DB, 부품 마스터 | typeahead 기준 데이터 |
| 관리 | 자료, 시트 전환, 설정, 감사 로그 | 운영 관리 |

### 5.2 사이드바 변경 요구

Notion 피드백에 따라 사이드바는 현재 1단계 메뉴 구조에서 1차-2차-3차-4차로 확장 가능한 구조로 변경한다.

요구사항:

- 사이드바는 1차 메뉴를 기준으로 접고 펼칠 수 있어야 한다.
- 2차 메뉴는 실제 화면이 있는 경우 해당 화면으로 이동한다.
- 3차/4차 메뉴는 아직 세부 화면 구상이 없어도 하위메뉴 또는 임시 화면으로 먼저 생성할 수 있어야 한다.
- 세부화면이 준비되지 않은 메뉴는 공란 화면, 준비중 화면, 또는 임시 리스트 화면으로 제공한다.
- 메뉴명은 개발 용어가 아니라 현장 업무 단어를 사용한다.
- 모바일 또는 좁은 화면에서는 메뉴가 아이콘 중심으로 접히거나 가로 스크롤 가능한 구조로 변환된다.

### 5.3 화면 이동 원칙

- 목록에서 항목을 클릭하면 새 페이지 이동보다 상세 패널 또는 모달을 우선 사용한다.
- 상세에서 다음 업무로 이어지는 버튼을 제공한다.
- `저장`, `작업으로 등록`, `완료`, `청구 생성`, `보증서 발행`은 명확히 분리한다.
- 사용자가 목록 맥락을 잃지 않도록 상세는 오른쪽 패널 또는 중앙 모달로 연다.

## 7. 검색과 입력 UX

### 7.1 검색형 선택

긴 목록은 라디오 버튼이나 긴 셀렉트 박스로 나열하지 않는다. 기본 패턴은 검색형 자동완성 선택이다.

적용 대상:

- 고객
- 차량번호
- 차대번호
- 거래처
- 정비소
- 보험사
- 보험 담당자
- 문의 경로
- 차종 DB
- 부품번호, 호환품번, 품명
- 재고 품목
- 작업 담당자 또는 출장자
- 수리부위, 옵션, 썬팅 항목

### 7.2 입력 원칙

- 처음 저장은 짧게 한다.
- 모르는 값 때문에 저장이 막히지 않아야 한다.
- 필수값과 선택값을 명확히 구분한다.
- 자동 계산값은 근거를 짧게 보여준다.
- 수동 수정한 값에는 수동 수정 표시와 이력을 남긴다.
- 다중 선택 항목은 칩 또는 라인 아이템으로 보여준다.
- 같은 값이 중복될 이유가 없는 옵션은 중복 선택을 막는다.
- 같은 부품을 여러 번 추가할 수 있는 업무는 수량/단가/메모가 있는 별도 라인으로 추가한다.

## 9. 비기능 요구사항

### 9.1 성능

- 페이지 로딩과 조작 반응은 매우 빨라야 한다.
- 최초 요구사항 기준으로 사용자는 3초 이상 대기하기 어렵다고 명시했다.
- 목록 검색과 typeahead는 인덱스 기반 검색을 적용한다.
- 고객 전화번호, 차량번호, 차대번호, 부품번호, 호환품번, 보험사명, 정비소명은 검색 인덱스를 둔다.
- 전화번호와 차량번호는 하이픈/공백 제거값으로도 검색 가능하게 한다.

### 9.2 안정성

- 서비스는 업무 시간 외에도 접근 가능해야 한다.
- 저장 실패, 발송 실패, 파일 생성 실패가 주요 업무 저장을 막지 않아야 한다.
- PDF/알림 발송은 별도 큐 또는 상태로 분리한다.
- 데이터 백업, 파일 백업, 장애 복구 기준을 운영 단계에서 정한다.

### 9.3 보안과 개인정보

- HTTPS를 적용한다.
- 비밀번호는 평문 저장하지 않는다.
- 연락처, 차량번호 등 개인정보성 데이터는 권한에 따라 마스킹할 수 있어야 한다.
- 접속 기록과 주요 변경 이력을 남긴다.
- 외부 시스템 계정, 운영 비밀번호, 실제 고객 데이터는 저장소와 문서에 남기지 않는다.
- 운영 단계에서는 개인정보보호법 준수 기준을 별도 점검한다.

### 9.4 반응형과 접근성

- PC 브라우저를 우선 지원한다.
- 반쪽 창과 작은 브라우저 창에서도 주요 업무가 가능해야 한다.
- 모바일에서는 사이드바와 상세 패널이 화면 폭에 맞게 접힌다.
- 버튼 텍스트는 잘리지 않아야 하며, 주요 버튼은 같은 위치에 반복 배치한다.
- 상태는 색상만으로 구분하지 않고 텍스트도 함께 표시한다.

- 기존 구글시트/엑셀 자료를 전체 이관할지, 필요한 기준정보만 먼저 이관할지
- 고객 전화번호, 차량번호, 차대번호의 마스킹 기준
- 취소/폐기 데이터 보존 기간

## 13. 수용 기준

MVP가 성공했다고 판단하기 위한 기준은 다음과 같다.

- 직원이 견적을 등록하고 작업으로 바로 전환할 수 있다.
- 작업자는 오늘 작업을 캘린더와 오늘 할 일에서 확인할 수 있다.
- 일정 클릭 시 작업 상세와 견적 내용을 확인할 수 있다.
- 작업 완료 후 사진과 사용 부품을 남길 수 있다.
- 작업 완료 후 사용 부품 기준으로 재고 출고 이력이 생성된다.
- 보험 청구와 고객 결제를 분리해서 관리할 수 있다.
- 카드매출 입금과 수수료를 별도 화면에서 확인할 수 있다.
- 대금결제리스트에서 반복 지급 일정을 확인할 수 있다.
- 보증서를 작업 또는 차량 기준으로 발행하고 다시 열람할 수 있다.
- 미수금 목록에서 받아야 할 금액을 바로 확인할 수 있다.
- 고객, 차량, 부품, 보험사, 정비소 검색이 반복 입력을 줄인다.
- 고객이나 거래처 상세에서 견적, 작업, 청구, 전표, 문서, 메모를 한눈에 확인할 수 있다.
- 주요 변경 이력이 남는다.
- 페이지 로딩과 조작 반응이 빠르며, 복잡한 화면에서도 입력이 끊기지 않는다.
