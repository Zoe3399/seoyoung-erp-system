# 프로젝트 작업 로그

이 문서는 포트폴리오와 회고를 위해 프로젝트 진행 과정을 누적 기록하는 문서다.

작성 원칙:

- 기능을 만들 때마다 왜 그렇게 설계했는지 함께 남긴다.
- 외부 시스템의 실제 데이터, 계정, 연락처, 주소 등 민감 정보는 기록하지 않는다.
- 코드 변경뿐 아니라 요구사항 분석, UI/UX 판단, 배포 판단도 기록한다.

## 현재까지의 주요 작업

### 1. 요구사항 문서 재검토

확인한 핵심:

- 기존에는 견적과 작업 스케줄을 구글시트로 관리한다.
- 목표는 간단한 조작으로 견적/작업내용을 등록, 관리, 검색하는 것이다.
- 핵심 기능은 로그인, 견적 등록, 작업 등록, 견적에서 작업으로 이동, 작업 캘린더다.
- 사용자는 내부 직원 중심이다.
- 페이지 로딩과 조작 반응이 빨라야 한다.
- 디자인은 심플하고 깔끔해야 한다.

설계 판단:

- 처음부터 대형 ERP 전체를 만들기보다 견적과 작업 Flow를 먼저 잡는다.
- 복잡한 필드는 나중에 보완할 수 있게 하고, 첫 저장을 쉽게 만든다.

### 2. 기존 글로벌 ERP 레퍼런스 분석

확인한 핵심:

- 견적, 작업, 청구, 재고, 부품 마스터, 차종 DB, 업체 관리가 핵심 도메인이다.
- 실제 자동차유리 업무에 필요한 필드가 많다.
- 청구와 재고 구조는 참고 가치가 높다.

설계 판단:

- 기존 시스템의 필드는 참고하되, UI는 그대로 따라가지 않는다.
- 레거시 관리자형 화면보다 내부 직원이 빠르게 쓰는 업무툴 형태로 재설계한다.
- 기존 시스템 데이터는 조회/분석 목적으로만 보고, 저장/수정/삭제하지 않는다.

### 3. MVP V2 설계 문서 작성

정리한 방향:

- 현재는 하나의 매장에서 사용하는 내부 시스템으로 본다.
- 여러 업체/다매장 구조는 고도화 단계로 둔다.
- 견적, 작업, 청구, 재고, 정산이 끊기지 않는 흐름을 우선한다.
- 자동차 브랜드, 모델명, 년식, 수리구분, 수리부위, 부품번호, 옵션, 썬팅 등은 선택값으로 둔다.
- 목록이 많아지는 필드는 typeahead 방식으로 설계한다.

### 4. 클라이언트 질문지 정리

정리한 방향:

- 지금 당장 레퍼런스나 회계적으로 판단 가능한 질문은 줄인다.
- IT를 모르는 클라이언트도 답변 가능한 쉬운 문장으로 정리한다.
- 구글시트에 복사해서 답변할 수 있는 형식도 함께 만든다.

### 5. 프로토타입 개발 시작

기술 방향:

- React + TypeScript + Vite
- 더미 데이터 기반 프론트엔드 프로토타입
- DB/API는 화면 방향 확정 후 연결

설계 판단:

- 빠르게 실행 가능한 프론트엔드 프로토타입을 먼저 만든다.
- 서버/DB보다 화면 Flow와 입력 UX를 먼저 검증한다.

### 6. 브랜드 변경 반영

변경 내용:

- 상호를 경남차유리로 변경
- 로고 SVG를 프로젝트 public 자산으로 복사
- 임시 favicon 추가
- 로그인/헤더 로고 영역 정리

설계 판단:

- 같은 상호가 여러 위치에 중복되지 않게 한다.
- 로그인 화면에는 회원가입을 제거하고 내부 직원 로그인 흐름으로 맞춘다.

### 7. 흰 화면 오류 수정

문제:

- 앱이 흰 화면으로만 보였다.

원인:

- React 앱을 실제 DOM에 mount하는 코드가 빠져 있었다.

조치:

- `createRoot(...).render(...)` 연결
- Vite 설정을 한국어 경로에서도 안정적으로 동작하도록 조정
- favicon 누락 이슈 정리

### 8. 작업 화면 UX 재설계

사용자 피드백:

- 고객 목록은 리스트 형태가 더 맞다.
- 클릭하면 상세는 모달/패널로 뜨는 게 좋다.
- 작업 캘린더는 카드 안 카드처럼 답답해 보이면 안 된다.
- 작업뿐 아니라 일반 할 일도 등록할 수 있어야 한다.

반영 내용:

- 작업 화면을 캘린더 + 오늘 할 일 조합으로 정리
- 일/주/월/년 보기 추가
- 작업과 일반 할 일을 함께 표시
- `작업 등록`과 `할 일 등록` 버튼 분리
- 일반 할 일은 색상으로 구분

### 9. 공통 UI 패턴 컴포넌트화

- `RecordToolbar`
- `SearchInput`
- `FilterTabs`
- `DetailDrawer`
- `ActionFooter`
- `FormSection`

적용 화면:

- 견적
- 청구
- 재고

설계 판단:

- 상세는 오른쪽 패널로 열어 목록 흐름을 끊지 않는다.
- 입력 폼은 구역별로 나누고 하단 버튼 영역을 공통화한다.

### 12. 대시보드 업무 바로가기 추가

사용자 요청:

- 대시보드에서 자주 쓰는 외부 링크를 직접 추가하고 싶다.
- 제목과 링크를 입력하면 제목을 클릭해서 이동할 수 있어야 한다.

반영 내용:

- 대시보드에 `업무 바로가기` 패널을 추가했다.
- 제목과 URL을 입력하면 바로가기 목록에 임시 추가된다.
- 제목을 클릭하면 새 탭으로 이동한다.
- 현재는 프로토타입 단계라 DB 저장은 하지 않고, 새로고침하면 초기값으로 돌아간다.

설계 판단:

- 실제 매장 업무에서는 보험사, 부품 주문, 세금계산서, 거래처 자료 등 외부 사이트를 자주 오갈 수 있다.
- 대시보드가 단순 요약 화면이 아니라 업무 시작점이 되려면 바로가기 영역이 필요하다.
- 저장형 기능은 백엔드 연결 이후 사용자별 설정으로 확장한다.

### 13. 클라이언트 검토용 상세 패널 Flow 연결

사용자 요청:

- 오늘 클라이언트와 검토할 수 있도록 클릭 시 모달/패널이 떠야 한다.
- 모바일 화면과 작은 브라우저 창에서도 작업하기 편해야 한다.
- 우선순위 1~8번 Flow를 먼저 프로토타입으로 확인하고 싶다.

반영 내용:

- 견적 목록에서 `보기`를 누르면 견적 상세 패널이 열린다.
- 견적 상세에서 `작업으로 등록`을 누르면 작업 등록 패널이 열린다.
- 작업 캘린더 카드와 오늘 할 일에서 상세/완료 Flow를 열 수 있게 했다.
- 작업 완료 패널에서 사진 첨부, 재고 차감, 청구 연결 단계를 한 화면에서 볼 수 있게 했다.
- 청구 목록에서 청구 상세와 입금 처리 미리보기를 열 수 있게 했다.
- 재고 목록에서 부품 상세와 입출고 이력 미리보기를 열 수 있게 했다.
- 자료 목록에서 자료 상세와 연결 업무 미리보기를 열 수 있게 했다.
- 반쪽 창과 모바일에서 상세 패널, 요약 카드, 입력 구역이 한 줄로 접히도록 반응형을 보강했다.

설계 판단:

- 실제 저장 기능보다 사용자가 클릭했을 때 다음 업무가 자연스럽게 이어지는지 확인하는 것이 우선이다.
- 상세는 새 페이지 이동보다 오른쪽 패널을 사용해 목록 맥락을 유지한다.
- 모바일과 작은 창에서는 패널이 화면 폭에 맞게 커지고, 그리드 항목은 한 줄로 접힌다.
- 현재 단계에서는 실제 DB 저장, 수정, 삭제 없이 더미 데이터 기반 화면 Flow만 확인한다.

### 14. 로그인 편의 기능과 계정/권한 V2 범위 정리

사용자 요청:

- 로그인 페이지에 ID 저장과 자동로그인을 추가한다.
- 아이디 찾기와 비밀번호 찾기는 당장 필요 없으므로 화면에서 제외한다.
- 계정 생성과 권한 관리는 이번 개발에서 바로 만들지 않고 V2 고도화로 둔다.

반영 내용:

- 로그인 화면에 ID 저장, 자동로그인 옵션을 추가했다.
- 아이디 찾기, 비밀번호 찾기 버튼은 주석 처리했다.
- `docs/mvp-v2-design.md`에 최고 관리자 기반 계정 생성, 직원/알바 계정, role 기반 권한 설계를 V2 계획으로 기록했다.

설계 판단:

- 이번 개발에서는 로그인 UX와 향후 확장 가능성만 반영한다.
- 실제 계정 생성, 비밀번호 초기화, 권한 변경은 최고 관리자 기능으로 묶어 V2 고도화에서 구현한다.
- 사용자에는 우선 `role` 필드를 두고, API와 프론트에서 `role -> permissions` 매핑을 기준으로 권한을 검사하는 방향을 고려한다.
